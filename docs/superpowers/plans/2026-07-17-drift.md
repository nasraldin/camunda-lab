# Drift detection (`camunda drift`) — Implementation Plan

> **For agentic workers:** Use superpowers:subagent-driven-development or executing-plans. Checkbox tracking. Commits only when asked. Requires env profiles + project paths.

**Goal:** Detect when git-tracked process assets diverge from what the active cluster has deployed — “Production differs: Process v12 vs Git v10”.

**Architecture:** Reuse local digests from `internal/plan` local inventory and remote definition versions/checksums. Optionally constrain to `git ls-files` under bpmn/dmn/forms so untracked local dirty files are called out separately.

**Tech Stack:** Go, git exec or go-git, env + HTTP clients shared with plan.

**Vision:** [../specs/2026-07-17-platform-toolkit-vision.md](../specs/2026-07-17-platform-toolkit-vision.md)

## Global Constraints

- Read-only toward cluster
- Exit `0` no drift, `1` drift detected, `2` error (CI-friendly)
- Distinguish: **git dirty** (working tree ≠ HEAD) vs **cluster drift** (HEAD or worktree ≠ cluster) via flags
- Default compare **working tree** to cluster; `--cached` compare HEAD to cluster
- Does not deploy or auto-sync

## Out of scope

- Auto `git pull` or cluster rollback
- Full Terraform state drift for Helm values (see `k8s` later)

## Success criteria

- CI can fail on exit `1` when cluster ≠ git
- Clear per-resource status lines
