# Kubernetes helpers (`camunda k8s`) — Implementation Plan

> **For agentic workers:** Use superpowers:subagent-driven-development or executing-plans. Checkbox tracking. Commits only when asked. Last among Phase 3 unless a Helm user needs it sooner.

**Goal:** Thin `kubectl` wrappers for common Camunda Helm release operations — same verb philosophy as lab compose ops, without replacing Helm or embedding cluster lifecycle.

**Architecture:** `internal/k8s` shells out to `kubectl` (must be on PATH) with label selectors for Camunda chart components. Flags for `--context`, `--namespace`, `--release`.

**Tech Stack:** Go, os/exec kubectl, Cobra.

**Vision:** [../specs/2026-07-17-platform-toolkit-vision.md](../specs/2026-07-17-platform-toolkit-vision.md)

## Global Constraints

- **Non-goal:** Installing Kind, creating clusters, or owning Helm values
- Require `kubectl` ; clear install hint if missing
- Do not talk to Docker Compose from these commands
- Prefer label selectors documented for Camunda 8 Helm charts; allow `--selector` override
- Destructive ops (`restart`) need `--yes`
- Complementarity: for deploy of BPMN to a K8s-hosted cluster, still official tooling — `k8s` is platform pod ops only

---

## Commands (MVP)

```bash
camunda k8s status
camunda k8s logs orchestration -f
camunda k8s restart connectors --yes
camunda k8s scale workers --replicas 3 --yes
```

| Subcommand | Behavior                                          |
| ---------- | ------------------------------------------------- |
| `status`   | `kubectl get pods,svc` filtered by release labels |
| `logs`     | `kubectl logs` for a component alias              |
| `restart`  | rollout restart deployment                        |
| `scale`    | scale deployment replicas                         |

Component aliases map to deployment name patterns (orchestration, connectors, optimize, identity, elasticsearch, …) with a single lookup table + override.
